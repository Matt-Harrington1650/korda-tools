# Korda Tools Windows Release and Update Guide

## Scope

This document defines the formal Windows release path for `korda-tools`.

- Windows is the only formal release target in this phase.
- `korda-tools` is the sole end-user distribution unit.
- Internal Sophon / RAG components are app-managed implementation details.
- Normal end users should install one Windows app, launch it from the Start menu or desktop, and use in-app diagnostics and update actions instead of shell commands.

## Official release architecture

- Source-of-truth version: `package.json`
- Formal release trigger: Git tag only
  - Stable: `vX.Y.Z`
  - Preview: `vX.Y.Z-rc.N`
- CI validation:
  - `main` and pull requests run `.github/workflows/ci.yml`
- Release publication:
  - `.github/workflows/release-windows.yml`
  - Builds with `tauri-apps/tauri-action`
  - Publishes to GitHub Releases
- Canonical installer:
  - Windows NSIS installer generated by Tauri
- Update channel:
  - Stable tags publish with `KORDA_RELEASE_CHANNEL=stable`
  - Release-candidate tags publish with `KORDA_RELEASE_CHANNEL=preview`
- Update transport:
  - HTTPS only
  - Default endpoint: `https://github.com/Matt-Harrington1650/korda-tools/releases/latest/download/latest.json`

## Versioning rules

- `package.json` is the authoritative version.
- `scripts/release/prepare-tauri-config.mjs` synchronizes:
  - `src-tauri/Cargo.toml`
  - `src-tauri/tauri.conf.json`
- `scripts/release/assert-release-tag.mjs` enforces:
  - pushed tag matches `package.json`
  - `CHANGELOG.md` contains a section for that version

Do not hand-edit Tauri or Cargo versions separately from `package.json`.

## Required repository configuration

### Repository variables

- `TAURI_UPDATER_PUBLIC_KEY`
  - Public updater verification key used by the app at runtime
- `TAURI_UPDATER_ENDPOINT` (optional)
  - Overrides the default GitHub Releases updater URL

### Repository secrets

- `TAURI_SIGNING_PRIVATE_KEY`
  - Private updater signing key used in CI to sign updater artifacts
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
  - Password protecting the updater private key

### Windows Authenticode secrets

Stable public releases require both of these:

- `WINDOWS_CERTIFICATE_BASE64`
  - Base64-encoded `.pfx` code-signing certificate
- `WINDOWS_CERTIFICATE_PASSWORD`
  - Password for the `.pfx`

Release-candidate tags may omit Windows Authenticode secrets. Updater signing is still required.

## Signing model

Two separate signing systems are used. Do not mix them up.

### 1. Tauri updater signing

Purpose:
- signs updater artifacts
- proves update authenticity to the installed app

Inputs:
- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
- `TAURI_UPDATER_PUBLIC_KEY`

This is mandatory for production updater releases.

### 2. Windows Authenticode signing

Purpose:
- signs Windows installer binaries for OS trust and publisher identity

Inputs:
- `WINDOWS_CERTIFICATE_BASE64`
- `WINDOWS_CERTIFICATE_PASSWORD`

This is wired through `scripts/release/sign-windows.ps1` and Tauri `signCommand`.

## Workflow behavior

### CI workflow

File:
- `.github/workflows/ci.yml`

Purpose:
- install dependencies
- sync release-aware Tauri config
- build frontend
- run tests
- run governance checks

This workflow does not publish releases.

### Release workflow

File:
- `.github/workflows/release-windows.yml`

Purpose:
- validate tag/version/changelog alignment
- validate updater signing inputs
- enforce Windows signing policy for stable releases
- render production Tauri config
- optionally import Windows code-signing material
- build and upload Windows release artifacts to GitHub Releases

The workflow fails fast when release-critical inputs are absent.

## Release procedure

### Stable production release

1. Update product changes.
2. Add a `CHANGELOG.md` entry for the target version.
3. Bump `package.json` version to the target SemVer.
4. Run:
   - `npm run version:sync`
   - `npm run typecheck`
   - `npm run test`
   - `npm run lint`
   - `cargo check` in `src-tauri`
5. Commit the release change.
6. Create and push the tag:
   - `git tag vX.Y.Z`
   - `git push origin vX.Y.Z`
7. GitHub Actions publishes the release.

### Preview release

1. Follow the same process with a prerelease version such as `0.2.0-rc.1`.
2. Tag with `v0.2.0-rc.1`.
3. Push the tag.

The workflow marks the GitHub Release as a prerelease and sets the in-app channel to `preview`.

## What the end user now does

### Install

1. Download the latest Korda Tools Windows installer from GitHub Releases.
2. Run the installer.
3. Launch Korda Tools from the Start menu or desktop shortcut.

### Launch and startup

On launch the app:
- prepares app data, logs, diagnostics, and temp directories
- starts the managed Sophon runtime worker
- checks readiness of managed local services
- attempts to start the local Docker-based runtime stack when required and allowed
- shows startup status in-app

The user does not need PowerShell for normal launch.

### Check for updates

Open:
- `Settings -> App Updates`

The app shows:
- current version
- release channel
- updater endpoint
- last checked time
- whether a newer release is available

### Install an update

1. Open `Settings -> App Updates`
2. Click `Check for Updates`
3. Click `Download and Install` when an update is available
4. Restart Korda Tools when prompted

## Startup and recovery UX

Open:
- `Settings -> Startup and Runtime`

The app exposes:
- overall startup state
- managed service health
- local app/log/temp paths
- last successful startup time
- retry button
- diagnostics export
- latest diagnostics path

The shell banner also surfaces degraded or blocked startup state on launch.

## Diagnostics and support

Diagnostics export writes a JSON bundle under the app log diagnostics directory and includes:
- release metadata
- startup status snapshot
- health probe summary
- runtime state snapshot when available
- `docker ps` summary when available
- recent app log excerpt

Use the in-app `Collect Diagnostics` action before escalating an incident.

## Internal vs public releases

### Internal/local builds

- `npm run tauri:dev` and local `tauri build` can run without updater signing secrets
- updater configuration is omitted when the public key is not present
- Windows Authenticode signing is skipped unless explicitly enabled

### Formal production releases

- require updater signing inputs
- require Windows Authenticode secrets for stable tags
- generate updater artifacts
- publish to GitHub Releases

## Clean Windows validation checklist

Validate on a clean Windows machine before broad distribution:

1. Install the NSIS package from GitHub Releases.
2. Launch without PowerShell.
3. Confirm startup status reaches `ready` or clearly explains a blocked dependency.
4. Confirm Settings shows:
   - correct version
   - correct channel
   - updater configured
5. Run `Check for Updates`.
6. Validate diagnostics export succeeds.
7. If validating an update:
   - install older release
   - publish newer release
   - confirm update is detected and installs

## Common release failures

### Tag does not match version

Symptom:
- release workflow fails in `Assert release tag + changelog alignment`

Fix:
- correct `package.json` or retag the release

### Missing updater signing material

Symptom:
- release workflow fails in `Validate updater signing inputs`

Fix:
- configure:
  - `TAURI_UPDATER_PUBLIC_KEY`
  - `TAURI_SIGNING_PRIVATE_KEY`
  - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`

### Stable release missing Windows certificate

Symptom:
- release workflow fails in `Validate Windows signing inputs`

Fix:
- configure:
  - `WINDOWS_CERTIFICATE_BASE64`
  - `WINDOWS_CERTIFICATE_PASSWORD`

### Updater not active in local build

Symptom:
- Settings shows updater unavailable

Cause:
- local/internal build did not provide `TAURI_UPDATER_PUBLIC_KEY`

Fix:
- expected for local builds unless you intentionally provide production-like updater env
